# 合同起草 — 领域知识

## 场景概述

合同起草根据业务需求自动生成合同文档。核心目标是减少人工起草工作量，同时确保合同条款合规。

## 用户上传材料处理策略

合同起草场景中，用户通常会上传以下材料作为合同生成的内容依据：

### 招投标文件（采购文件）
用户上传的 docx/pdf 文件，通过文件工具读取后提取以下关键信息：
- **项目名称**：作为合同名称的基础
- **项目背景与依据**：写入合同背景条款（如政策文件引用）
- **技术要求**：作为合同标的的功能描述和验收标准
- **资质要求**：用于供应商资质条款
- **服务要求**：完工期限、质保期、响应时效等写入相应条款
- **预算金额**：用于校验合同金额是否超预算
- **付款方式**：写入付款条款
- **评审标准**：归档备查，不直接写入合同

### 中标通知书
用户上传的 docx/pdf 文件，通过文件工具读取后提取以下关键信息：
- **成交供应商**：乙方名称、法定代表人、地址、统一社会信用代码
- **成交金额**：作为合同总金额（大小写必须一致）
- **产品明细**：品名、品牌、型号、数量、单价 → 写入合同标的表
- **主要条款**：完工期限、质保期限、付款比例、违约金比例
- **知识产权约定**：写入知识产权条款

### 材料处理流程
1. 用户上传材料 → Agent 使用 `read_file` 工具读取文件内容
2. 从招投标文件提取：项目背景、技术要求、服务要求、付款方式
3. 从中标通知书提取：供应商信息、成交金额、产品明细、合同条款
4. 通过 `get_supplier_info` 补充供应商资质和信用评级
5. 通过 `get_contract_template` 获取合同模板
6. 将提取的信息填充到模板中生成合同

**注意**：中标通知书中的金额和条款是最终确认的，优先级高于招投标文件中的预算和初始要求。

## 合同字段分类与填写策略

### 1. 甲方信息（通常为 known_values）
- 甲方全称、统一社会信用代码、法定代表人、联系地址
- 这些字段由 HRP 系统推送，Agent **不可修改**

### 2. 乙方信息（从中标通知书或用户材料提取）
- 乙方全称：从中标通知书提取
- 法定代表人、地址、统一社会信用代码：从中标通知书提取
- 供应商资质和信用评级：通过 `get_supplier_info` 查询补充
- 无法确定时标记为 [待填写]

### 3. 合同标的条款
- 品名/服务内容：从中标通知书产品明细或招投标文件技术要求中提取
- 规格型号、数量、单价：从中标通知书产品明细中提取

### 4. 金额条款
- 合同总金额：大写和小写必须一致
- 使用 calculator 工具验证金额计算
- 付款方式：预付款比例不超过 30%
- 付款节点：按合同类型设置标准节点

### 5. 期限条款
- 合同有效期：起止日期
- 交付/服务期限、质保期限
- 使用 `date_diff` 校验日期合理性

### 6. 法律条款（通常使用模板默认值）
- 违约责任：违约金比例通常 5%-20%
- 争议解决方式、保密条款、知识产权归属

## 合同文档结构

生成合同文档（document_type: contract_text）时应包含：
- 合同名称、编号
- 合同双方信息
- 合同标的（产品/服务描述）
- 合同金额与付款方式
- 交付/服务期限与验收标准
- 质量保证
- 违约责任
- 保密条款
- 争议解决
- 合同生效条件
- 附件清单

**文档规范**：
- 未填字段用 `[待填写]` 占位
- 金额同时使用大写和小写（如：人民币伍拾万元整（¥500,000.00））
- 日期格式：YYYY年MM月DD日
- 法律条款使用规范法律用语

## 自检审计知识

合同生成后建议进行自检，检查以下方面：

### 必备条款完整性
- 合同双方信息是否完整
- 合同标的是否清晰明确
- 金额和付款方式是否明确
- 违约条款是否存在

### 内部一致性
- 金额大小写是否一致（使用 calculator 验证）
- 各付款节点金额之和是否等于总金额（使用 `sum_values` + `numeric_compare`）
- 日期逻辑是否合理（起始日期 < 结束日期）

### 合规性预检
- 预付款比例是否超过 30%（使用 `calculate_ratio`）
- 违约金比例是否在 5%-20% 范围内
- 合同金额是否匹配审批权限（10万/50万/100万分级）

## 质量保障要点

1. **条款完整性**: 合同文档必须包含所有必备条款，缺少的用 [待填写] 标记
2. **数值准确性**: 金额计算、比例计算必须使用 calculator 工具
3. **已知值保护**: 甲方信息等 known_values 严格不可修改
4. **法律规范性**: 法律条款用语准确规范
